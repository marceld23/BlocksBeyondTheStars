# AI Backend â€” Decision

Covers the optional AI mission system and the LLM text endpoints: mission plans (L0), NPC greetings
(L1/L2), board-mission texts (L3) and ship-AI (VEGA) banter. Endpoint details live in
[ai-backend/README.md](../ai-backend/README.md).

## Decision

- The AI is an **optional, separate Python service** (`ai-backend/`, FastAPI). The game
  server is C#/.NET and **stays authoritative**.
- The first mission MVP works **entirely without AI** (already implemented in M13). AI is
  additive and **off by default** (`AiLevel = Off`).
- The AI **proposes** a structured `MissionPlan` (creative layer + rule layer). The server
  **validates, clamps and publishes** it. The AI never grants items, finalizes rewards or
  bypasses rules.

## Levels (`AiLevel` in server config)

| Level | Behaviour |
|---|---|
| `Off` | Server never contacts the backend (default). |
| `TextOnly` | (Reserved) AI only writes text for existing missions; no full generation. |
| `Suggest` | Valid AI missions are stored as **inactive drafts** for admin review. |
| `Auto` | Valid AI missions are **published** immediately. |

Any level **other than `Off`** also lets the server fetch the flavour-text endpoints: **NPC greetings**
(`/npc-line`), **ship-AI (VEGA) banter** (same endpoint, role `ship_ai`) and **board-mission texts**
(`/mission-text` â€” Title/Description written around the FIXED server-coined job; objective/reward untouched).

## Flow

1. An admin triggers generation (admin command `ai_mission` with a context string).
2. Server asks the backend (`POST /mission-plan`), enriching the context with the ALLOWED content keys
   (mineable targets + reward items) so the LLM can't hallucinate keys the validator rejects; on any
   error it logs and falls back to "no mission" â€” never crashes.
3. Server converts `MissionPlan` â†’ `MissionDefinition`, **clamping reward counts (â‰¤25)**.
4. `MissionValidator` checks objectives/rewards against loaded content; invalid â†’ rejected.
5. Per level: `Auto` publishes, `Suggest` drafts (inactive).

## Why this shape

- Keeps the heavy/optional generative dependency out of the lightweight, Pi-friendly game
  server (separate process, separate language).
- Server-side validation + reward clamping make AI output safe by construction.
- A real LLM can replace the template generator behind the same HTTP contract with no
  server change.

## NPC greetings (item 15)

A second endpoint, `POST /npc-line`, returns a short in-character greeting (`{ "Text": "â€¦" }`) for a
settlement/station NPC, generated by an **LLM via LangChain + LangGraph** when configured, else a
deterministic **bilingual template**. It is **provider-agnostic** through the OpenAI-compatible chat API, so the
same backend works with **LM Studio (self-hosted) / OpenAI / Claude** by env (see `ai-backend/.env.example`).
The request also carries **persona + recent-events memory** (L2) so the NPC's voice is stable per NPC and
the line can reference past interactions; with role `ship_ai` the same endpoint serves rare **VEGA banter**
(cached per world/day-phase/locale; absent when AI is off â€” scripted VEGA lines cover everything else).

Flow (server side, `GameServerNpcGreeting.cs`):

1. The player opens an interaction â€” a vendor's market (client `NpcGreetIntent`) or a mission board
   (server-side in `SendMissionList`). The server picks the nearest in-reach NPC of that role (proximity-gated).
2. It builds context from the **playerâ†”NPC relationship memory** (score + tier + past visits), the NPC
   (name/role/theme/android), the settlement, and the player's **locale**, then asks the backend.
3. Generation runs **off the game thread** (never blocks the tick) and the line is **cached** by
   `npcKey|locale|relationship-tier`; a 25 s per-NPC cooldown stops re-open spam.
4. The line is sent as an `NpcGreeting` and shown as a **speech bubble** over the NPC. When `AiLevel = Off` or
   the backend is unreachable the server sends an empty line and the **client renders a localized static
   fallback** (`npc.greet.*`, DE+EN), so greetings always appear. Greetings are flavour only â€” never gameplay.
